# chatMind 代码与提示词工程优化说明

本次优化针对 HAMD V3 评估主链路：`15.2_SenceVoice_QWen2.5_HAMD_V3.py`（语音交互主程序）和 `hamd_evaluator_v3.py`（评估器）。所有改动已通过离线冒烟测试（mock LLM 验证三阶段门控、追问累积、高风险触发、报告生成四条路径）。

---

## 一、性能优化：每轮对话的 LLM/TTS 调用次数减半以上

**原流程**（用户每答一题）：

| 步骤 | 调用 |
|---|---|
| 情绪分析 | LLM 调用 ①（独立 prompt，返回 JSON） |
| 清晰度评估 | LLM 调用 ②（独立 prompt，返回小数） |
| 评分 | LLM 调用 ③（独立 prompt，返回"评分：X分"） |
| 共情回应 | LLM 调用 ④ + TTS 播报 ① |
| 生成下一题 | LLM 调用 ⑤ + TTS 播报 ② |

合计 **5 次串行 LLM 调用 + 2 次 TTS**，实时语音场景下用户每答一句要等很久。

**新流程**：

1. **情绪 + 清晰度 + 评分合并为单次结构化调用**（`analyze_answer_combined`）：一个 prompt 同时输出 `{score, clarity, emotion, intensity, rationale}` JSON。三个判断共享同一份"题目 + 评分标准 + 回答 + 上下文"输入，合并后不仅省 2 次调用，判断之间还天然一致（如"答非所问 → score=0 且 clarity 低"）。
2. **共情回应与下一题合并为一次生成、一次播报**：`generate_dynamic_question` 新增"共情过渡模式"，prompt 要求"先用一句不超过 12 字的共情回应，再自然过渡到新问题"，省 1 次 LLM 调用和 1 次 TTS 往返，且语音听感上比"两段分开播"更自然。

合计降为 **2 次 LLM 调用 + 1 次 TTS**（追问轮次只需 2 次 LLM + 1 次 TTS，同前）。

另外：OpenAI client 改为模块级单例（原先每次调用都 `OpenAI(...)` 重建连接），并加入失败重试（2 次，退避 0.5s/1s）。

## 二、功能修复：三阶段门控原来是"死代码"

`stage_thresholds = {1: 5, 2: 10}` 在原代码中定义了但**从未被使用**——`get_current_question` 无条件自动进入下一阶段，所以"初筛 → 深入 → 全面"的自适应设计实际上不存在，所有用户都要答完 16 题。

现已实现真正的门控（`_should_enter_next_stage`）：

- 阶段 1 结束后累计得分 ≥ 5 才进入阶段 2，否则提前结束（初筛阴性，约 5 题、3 分钟完成）；
- 阶段 2 结束后累计得分 ≥ 10 才进入阶段 3；
- **安全兜底**：阶段 1 即使分数未达阈值，只要检测到高风险信号（危机情绪 / risk_level 为 high/critical），强制进入阶段 2 完成自杀风险条目，防止"初筛分低但有危机表达"的用户被漏掉。

配套修改：

- 提前结束时报告标注 `early_terminated: true`，severity 附加说明"总分基于已作答条目，仅供初筛参考"，避免"5 题总分低 = 完整量表正常"的误读；
- 分项分析只统计实际作答条目，不再把未答题按 0 分稀释百分比；
- 报告新增 `answered_questions / total_questions` 字段。

## 三、评分有效性修复

1. **追问回答不再被丢弃**：原逻辑追问后只对用户最后一句话评分，首答信息全部丢失。现在 `current_answer_parts` 累积本题所有回答（首答 + 追问补充），评分基于合并后的完整信息（冒烟测试场景 3 验证）。
2. **评分提示词对齐"实际问的问题"**：原评分 prompt 用题库的原始问题文本，但用户实际听到的是 LLM 动态生成的问题，问答错位会影响评分。现在记录 `last_asked_question` 并传入评分 prompt（`【实际提问】`字段）。
3. **评分 prompt 加入防夸大约束**："只依据回答中的事实信息（频率、持续时间、对功能的影响），不要因情绪化用词而夸大评分"——LLM 评分员容易被强烈措辞带偏，这是已知的评分者偏差。
4. 判别类任务统一低温采样：综合分析 temperature=0.2、意图分类 temperature=0.1（原意图分类用默认 0.7，分类结果不稳定）。

## 四、健壮性修复（bug 清单）

| # | 问题 | 修复 |
|---|---|---|
| 1 | LLM 返回 ```json 围栏或带前后缀文字时 `json.loads` 直接抛异常 | 新增 `_extract_json`：剥围栏、提取首个 `{...}` 块再解析；意图分类同样处理 |
| 2 | `generate_personalized_feedback` 在 `level` 赋值前出异常时，except 分支引用未定义的 `level` 再次 NameError | `level` 提前用 `.get` 安全取值 |
| 3 | **"没有"（2 字）等有效回答被最短长度过滤丢弃**——自杀题答"没有"系统会沉默不响应 | 增加 `SHORT_VALID_ANSWERS` 白名单（没有/有/是的/不会/偶尔/经常……） |
| 4 | API 最终失败时返回的道歉文案"抱歉，我现在无法连接……"长度恰好通过校验，会被当作"生成的问题"或"评分结果"使用 | `chat_api` 增加 `fallback` 参数，评估器路径失败返回空串，触发评估器内部的规则兜底 |
| 5 | pygame mixer 在录音线程、播放函数中反复 `init()/quit()`，多线程下易导致音频设备异常 | 模块级初始化一次，播放函数不再 init/quit |

| 7 | `cv2`、`numpy` import 了但全文未使用（白白依赖 OpenCV） | 移除 |
| 8 | 评分字段重复解析逻辑分散、输出含问号时整体丢弃为 0 分 | 结构化 JSON 输出后统一做边界约束（score 钳到 0-4，clarity 钳到 0-1，emotion 枚举校验） |

## 五、提示词工程改进要点（汇总）

1. **结构化输出**：所有判别任务（评分/情绪/清晰度/意图）统一要求"只输出一个 JSON 对象，禁止 markdown 围栏"，并在代码侧做容错解析 + 字段钳制，双保险。
2. **角色精确化**：评分角色从"心理评估评分机器人"改为"精神科量表评定员"，并明确评分只能依据"频率、持续时间、功能影响"等事实维度。
3. **字段语义内嵌于 schema**：clarity 各档位（1.0/0.7/0.4/0.2）的判据直接写在 JSON 字段说明里，而不是散落在正文。
4. **共情与提问合并生成**，并约束"共情部分禁止提问，整段只能有结尾一个问句"，避免 TTS 播出双问句让用户困惑。
5. **上下文用途声明**：对话摘要标注"仅用于理解语境，不直接参与本条目评分"，防止 LLM 把上一题的严重程度迁移到本题。
6. **温度分层**：生成类任务（提问、反馈）0.7–0.8，判别类任务 0.1–0.3。

## 六、遗留问题（建议后续讨论，本次未改动）

1. **量表心理测量学问题**：本实现是 16 条目、每条 0–4 分（满分 64）的 HAMD 变体，但严重程度切分（≤7 正常 / ≤17 轻度 / ≤23 中度）沿用的是 HAMD-17 的常用分界，而标准 HAMD-17 中有 9 个条目只计 0–2 分（满分 52）。两者刻度不一致，分界线严格来说不可直接套用。若用于论文，建议：要么按标准 HAMD-17 条目和计分实现，要么把当前版本明确表述为"HAMD 启发的自适应筛查工具"并自行标定分界。
2. **LLM 评分一致性未标定**：建议用若干标注样本测 LLM 评分与人工评分的一致性（如加权 Kappa / ICC），这是评审一定会问的点。
3. 主程序的线程同步用的是布尔标志位（`hamd_processing_lock`），CPython 下基本够用但不严格；如后续重构可换 `threading.Lock`。
4. `webrtcvad` 检测使用 0.02s 帧步长逐帧投票，参数（VAD_MODE=3、0.5 投票率）可针对实际麦克风环境调优。

# 第二轮：面向市场化的生产级改造（参考 智沃灵引 SOUL）

参考对标产品 [soul.zhiwomind.com](https://soul.zhiwomind.com/)（智沃灵引 SOUL，AI 心理咨询平台）的产品设计，对系统做了从"实验室 demo"到"可投入真实使用"的改造。借鉴的核心设计：情绪签到、知情同意、隐私数据用完即毁、统一危机协议、结构化报告建议卡片、温暖专业的产品文案。

## 七、评估流程升级：触发 → 知情同意 → 情绪签到 → 答题

原流程：说"心理评估"直接开始问第一题。新流程（`hamd_phase` 状态机）：

1. **知情同意**（合规必备，可配置关闭）：开始前播报——这是自评不是诊断、数据只存本机、随时可说"停止评估"退出。用户说"同意/开始"才进入；说"算了/不做"温和退出；听不清则再确认一次。
2. **情绪签到**（借鉴 SOUL"此刻的你，感觉如何？"）：先问当下感受，规则匹配 6 类情绪（平静/开心/困惑/焦虑/难过/愤怒），每类配一句共情短句即时播报（同时掩盖首题 LLM 生成的等待延迟），并把签到结果注入评估器作为**情绪基线**——签到时就情绪强烈的用户，风险等级和提问语气从第一题起就是校准过的。
3. 签到/同意阶段说出危机表达（"不想活了"）→ 立即危机响应，不会被当作普通回答处理。

## 八、统一危机协议（原来普通聊天模式完全没有危机处理）

- 新增 `check_crisis()` 规则检测（零延迟，不依赖 LLM），**普通对话模式、同意阶段、签到阶段、答题阶段全程生效**；
- 统一危机话术 `crisis_response()`：表达关心 + 全国心理援助热线 12356 + 鼓励联系信任的人 + 不中断对话；
- 热线号码进入配置文件，评估报告中的危机建议同步使用配置的号码（原代码硬编码 400-161-9995）。

## 九、隐私保护（对标 SOUL"访客倾诉：会话结束数据自动销毁"）

- 录音文件 `output/audio_N.wav` 原来无限累积——现在 ASR 用完即删（`delete_audio_after_use`，可配置）；
- TTS 临时 mp3 播放完即删（`delete_tts_after_play`）；
- 评估报告仍保存本机 `HAMD_Results/`（知情同意话术中已说明）。

## 十、人设与产品文案重写

- 普通对话人设从"18 岁活泼俏皮女大学生"改为"温暖、专业的心理健康陪伴助手"（市场化心理产品不能用娱乐化人设承接抑郁评估入口），提示词明确：共情优先、口语化适合语音播报、不诊断不荐药、困扰时引导自评、危机时不转移话题；
- 开场白、同意话术、签到文案按 SOUL 的"温柔接纳"风格重写（自写文案，未照抄）。

## 十一、工程化

| 项 | 改造 |
|---|---|
| 配置外置 | 新增 `chatmind_config.json`：模型、TTS 音色、VAD 停顿、阶段阈值、追问上限、危机热线/关键词、隐私开关、助手名称；缺失项回落默认值 |
| 日志系统 | `logging` 双通道（控制台 + `./logs/chatmind.log`），关键事件（ASR、播报、评分、危机告警、评估完成）带时间戳可审计 |
| 推理线程异常保护 | `Inference` 包装层捕获一切异常并播报兜底话术——原来线程内异常会让系统对当前轮静默 |
| TTS 失败降级 | edge-tts 依赖网络，失败时降级为日志文本输出，不再静默卡死 |
| 评估器参数化 | `stage_thresholds`/`max_follow_up`/`crisis_hotline` 通过构造函数注入，便于 A/B 调参 |
| 空报告保护 | 刚开始就退出时 `generate_report` 返回 error，完成函数不再 KeyError |

## 十二、报告升级（对标 SOUL 报告页）

新增字段：`structured_suggestions`（建议卡片数组，title/description/frequency 三元组，按严重程度分级：正常→保持节律/适度运动/社交联结；轻度→正念冥想/情绪日记/适度运动；中度→专业帮助/社会支持/温和活动；重度→及时就医/危机支持/不要独处），`conversation_turns`（对话轮数）。配合已有的时长、情绪汇总、分项分析，报告结构已与 SOUL 的"情绪分析报告"页对齐，可直接对接前端渲染。

## 十三、距离真正上市还需要什么（本次未覆盖，需团队决策）

1. **资质与合规**：心理类产品涉及《互联网诊疗管理办法》边界问题，需法务评估"自评工具"定位；隐私协议文本（可参考 SOUL 的隐私协议结构：收集范围/使用/共享/存储/用户权利/未成年人）；
2. **架构**：当前是单机单用户脚本，市场化需服务端化（会话管理、用户体系、报告云存储加密）——SOUL 采用 Web SPA + 后端 API + JWT 的标准架构可作参考；
3. **未成年人保护**：SOUL 明确"主要面向成年人"，本系统应在同意环节增加年龄确认；
4. **LLM 评分一致性标定**（与第一轮遗留问题相同）；
5. **量表分界标定**（与第一轮遗留问题相同）。

# 第三轮：隐式对话式评测 + 代码结构重组

## 十四、评测体验从"显式问答"改为"隐式对话"

旧体验：说"心理评估" → "好的，我将为您进行心理评估" → 知情同意宣读 → 逐题提问 → "评估完成，您的结果是……"。问卷感强，用户处于"被测试"状态，回答易受社会期许偏差影响。

新体验（`assessment.style: "implicit"`）：

- **触发仍由用户主动发起**（"帮我做个评测/聊聊我最近的状态"）——这保证了知情前提；开场只有一句轻量透明告知（"像平时一样随便聊聊，想停直接说"），没有同意仪式；
- **对话即评估**：每轮回应由 LLM 以"朋友接话"方式生成——先接住上一句，再自然聊到下一个 HAMD 维度；评分/清晰度/情绪在后台单次结构化调用静默完成；
- **提示词硬约束（反问卷规则）**：禁止出现"评估、测试、题目、量表、问卷、打分、第几个问题"等字眼；禁止医生问诊腔；一段话只许结尾一个问句；禁止说教和安慰套话。冒烟测试对全程播报做了禁词断言；
- **自然退出**："先这样吧/换个话题"即结束，已聊部分仍保存报告；
- **收尾不报分数**：LLM 生成观察式反馈（"听下来你最近确实挺累的"），不贴标签；中重度自然引导专业求助；报告 JSON 静默保存。

伦理设计说明：隐式的是"体验"，不是"评估行为本身"——评测必须由用户主动发起，开场有透明告知，绝不在普通聊天中偷偷打分。这条边界写进了代码注释和本文档。

## 十五、文件结构重组（单体脚本 → 包）

原 `15.2_*.py` 是 700 行单体脚本（录音/VAD/ASR/LLM/TTS/评估编排/全局状态混在一起）。现拆为：

```
main.py                  入口装配（~100行）
chatmind/config.py       配置与日志
chatmind/llm_client.py   LLM客户端
chatmind/speech.py       TTS + ASR
chatmind/audio_capture.py 采集与VAD（类封装，回调注入解耦）
chatmind/evaluator.py    评估器（隐式提示词版）
chatmind/dialogue.py     对话编排状态机（idle/assessing）
```

关键解耦：`AudioCapture` 通过注入的 `pause_threshold_fn`（评测等待时停顿阈值放宽）和 `should_ignore_fn`（播报/处理中丢弃语音）与上层协作，不再共享全局变量。旧脚本保留为参考（README 已标注）。

## 十六、其它更新

- README 全面重写：隐式流程示例对话、新结构、修正了旧版缺失的依赖（`openai`、`pypinyin` 旧 README 根本没列）、方法学免责声明；评分细则改为指向 `hamd_questions_v3.json` 单一来源；
- 收尾反馈提示词：向 LLM 提供 severity/risk 标签仅用于组织语言，明确"绝不能向用户透露这些标签"；
- 危机协议在隐式模式下保持显式——这是唯一允许打破"自然聊天"的场景（安全 > 体验）。
